# CLAUDE.md — The Complete Guide

## What CLAUDE.md Is

CLAUDE.md is loaded at the start of every Claude Code session. It's Claude's persistent
memory about your project. It answers: "What does Claude need to know to work on this
codebase without asking me?"

## The 200-Line Rule

Keep CLAUDE.md under 200 lines. Every line beyond that reduces adherence to ALL lines.
If Claude ignores your instructions, the file is probably too long. Trim ruthlessly.

**What belongs in CLAUDE.md:** Things Claude needs to know EVERY session.
**What doesn't belong:** Things Claude only needs sometimes (put those in skills).

## Optimal Structure

```markdown
# [Project Name]

## Quick Facts
- **Stack**: [languages, frameworks, key libraries]
- **Node/Python version**: [version]
- **Package manager**: [npm/yarn/pnpm/pip/cargo]
- **Test command**: `[command]`
- **Lint command**: `[command]`
- **Build command**: `[command]`
- **Dev command**: `[command]`

## Architecture
[2-5 sentences about how the project is organized]
- `src/` — [what's here]
- `tests/` — [what's here]
- `[other key dirs]` — [what's here]

## Code Conventions
- [Convention 1 — be specific]
- [Convention 2 — be specific]
- [Convention 3 — be specific]

## Common Gotchas
- [Thing that trips up Claude repeatedly]
- [Another thing]

## What NOT to Do
- NEVER [specific bad behavior Claude keeps doing]
- NEVER [another one]
```

## Section-by-Section Guidance

### Quick Facts

Include ONLY what Claude needs to run commands. Not your entire tech stack history.

| Include | Don't Include |
|---------|-------------|
| `npm run test` | Which testing framework and why you chose it |
| `Python 3.11+` | Your Python version upgrade history |
| `pnpm` as package manager | Why you switched from yarn |
| `npm run lint` | Your ESLint config philosophy |

### Architecture

Maximum 5 sentences + a directory listing of KEY directories (not every folder).

**Good:**
```
Monorepo with 3 packages: core (shared logic), web (Next.js frontend), api (Express backend).
Packages communicate via shared TypeScript types in core/types/.
Database is PostgreSQL via Prisma ORM. Migrations in api/prisma/.
```

**Bad:**
```
This project was started in 2023 when we decided to migrate from our legacy
PHP codebase to a modern JavaScript stack. We evaluated several frameworks
including Remix, SvelteKit, and Next.js before settling on Next.js due to...
[200 words of context Claude doesn't need]
```

### Code Conventions

Be specific and testable. Vague rules get ignored.

| Vague (gets ignored) | Specific (gets followed) |
|---------------------|------------------------|
| "Write clean code" | "Functions must be under 30 lines" |
| "Follow best practices" | "Use `async/await`, never raw Promises" |
| "Write good tests" | "Every exported function needs a test in `__tests__/`" |
| "Use TypeScript properly" | "No `any` type — use `unknown` with type guards" |

### Common Gotchas

These are the things you've had to correct Claude on more than once.

**Examples:**
```
- This project uses `src/` for source files, not `lib/`. Don't create files in lib/.
- Database columns are snake_case, TypeScript properties are camelCase. The ORM handles conversion.
- Environment variables are in .env.local (gitignored), not .env. Never write secrets to .env.
- The `utils/` directory is for pure functions only. Side effects go in `services/`.
```

### What NOT to Do

Use CAPS and strong language here. These are the critical guardrails.

```
- NEVER modify files in `generated/` — these are auto-generated by the schema.
- NEVER run `npm install` without asking first — we use exact versions.
- NEVER commit directly to main — always create a branch.
- NEVER delete test files, even if they seem unused.
```

### React / Next.js

```markdown
# [Name]

## Quick Facts
- **Stack**: Next.js 15, React 19, TypeScript, Tailwind CSS
- **Package manager**: pnpm
- **Dev**: `pnpm dev` (port 3000)
- **Build**: `pnpm build`
- **Test**: `pnpm test` (vitest + testing-library)
- **Lint**: `pnpm lint`

## Architecture
- `app/` — App Router pages and layouts
- `components/` — Reusable UI components
- `lib/` — Utilities and helpers
- `actions/` — Server actions

## Conventions
- Server Components by default, `'use client'` only when needed
- Tailwind for styling, no CSS modules
- Zod for form validation
- Server Actions for mutations, not API routes

## Gotchas
- Next.js 15 uses async `cookies()` and `headers()` — don't forget `await`
- `useRouter` from `next/navigation`, not `next/router`
```

## What Goes in CLAUDE.md vs Elsewhere

| Information | Where | Why |
|------------|-------|-----|
| Test command | CLAUDE.md | Needed every session |
| Coding conventions | CLAUDE.md | Needed every session |
| API design patterns | Skill | Only when building APIs |
| Deployment process | Command (`/deploy`) | Only when deploying |
| Code review checklist | Agent (`code-reviewer`) | Only when reviewing |
| Database migration guide | Skill | Only when migrating |
| Onboarding for new devs | Command (`/onboard`) | Only when onboarding |

## CLAUDE.md Anti-Patterns

| Anti-Pattern | Problem | Fix |
|-------------|---------|-----|
| 500+ lines | Claude ignores most of it | Cut to < 200 lines |
| Paragraph-style instructions | Hard to scan | Use bullet points |
| "Please" and "try to" | Weak compliance | Use "MUST" and "NEVER" |
| Duplicate of README | Wastes tokens on non-actionable info | CLAUDE.md ≠ README |
| Full API documentation | Too much context | Put in a skill |
| History/changelog | Irrelevant to coding | Remove |
| Links to external docs | Claude can't browse | Inline the key points |
