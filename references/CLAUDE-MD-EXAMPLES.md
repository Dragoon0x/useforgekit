## 1. TypeScript Library (npm package)

```markdown
# my-lib

## Quick Facts
- **Stack**: TypeScript 5.5, Node.js 22
- **Build**: `pnpm build` (tsup â†’ ESM + CJS)
- **Test**: `pnpm test` (vitest)
- **Lint**: `pnpm lint` (eslint + prettier)
- **Publish**: `npm publish` (after build + test)

## Architecture
Single-package library. Source in `src/`, tests co-located as `*.test.ts`.
Entry point: `src/index.ts`. Public API is everything exported from index.

## Conventions
- TypeScript strict mode. No `any`. Use `unknown` with type guards.
- Named exports only. No default exports.
- Functions under 30 lines. Extract if longer.
- Every exported function has a JSDoc comment with @param and @returns.
- Error messages include the invalid value: `Expected string, got ${typeof input}`

## What NOT to Do
- NEVER add dependencies without asking. This is a zero-dep library.
- NEVER modify `dist/` â€” it's generated by build.
- NEVER use `console.log` in library code. Throw errors instead.
- NEVER change the public API without bumping major version.
```

## 2. Next.js 15 App

```markdown
# my-app

## Quick Facts
- **Stack**: Next.js 15, React 19, TypeScript, Tailwind CSS 4, Drizzle ORM
- **DB**: PostgreSQL via Drizzle (`drizzle-kit push` for schema changes)
- **Dev**: `pnpm dev` (port 3000)
- **Build**: `pnpm build`

## Architecture
- `app/` â€” App Router. Layouts, pages, loading/error states.
- `app/api/` â€” API routes (prefer Server Actions over these).
- `components/` â€” Reusable UI. Each component gets its own folder.
- `lib/` â€” Utilities, DB client, auth helpers.
- `drizzle/` â€” Schema definitions and migrations.

## Conventions
- Server Components by default. Add `'use client'` only for interactivity.
- Server Actions for mutations. API routes only for webhooks/external calls.
- Tailwind for ALL styling. No CSS modules, no styled-components.
- Zod for form validation (shared between client and server).
- `next/navigation` for routing. NEVER `next/router`.
- Async `cookies()` and `headers()` â€” always `await`.

## What NOT to Do
- NEVER use `pages/` directory. This project uses App Router only.
- NEVER install CSS libraries (no MUI, no Chakra). Use Tailwind.
- NEVER put secrets in `.env`. Use `.env.local` (gitignored).
- NEVER use `getServerSideProps` or `getStaticProps`. These are Pages Router.
```

## 3. Python FastAPI Backend

```markdown
# my-api

## Quick Facts
- **Stack**: Python 3.12, FastAPI, SQLAlchemy 2, Pydantic v2
- **Package manager**: uv
- **Dev**: `uv run uvicorn app.main:app --reload`
- **Test**: `uv run pytest -x`
- **Lint**: `uv run ruff check .`
- **Format**: `uv run ruff format .`
- **Type check**: `uv run mypy app/`
- **Migrations**: `uv run alembic upgrade head`

## Architecture
- `app/` â€” Application code
- `app/main.py` â€” FastAPI app instance, router includes
- `app/routers/` â€” Route handlers grouped by domain
- `app/models/` â€” SQLAlchemy ORM models
- `app/schemas/` â€” Pydantic request/response schemas
- `app/services/` â€” Business logic (NOT in routers)
- `app/deps.py` â€” Dependency injection (DB session, current user)
- `tests/` â€” Mirrors app/ structure

## Conventions
- Type hints on ALL function signatures. No untyped functions.
- Pydantic models for ALL request/response bodies. Never raw dicts.
- Business logic in `services/`, NOT in route handlers.
- Route handlers are thin: validate â†’ call service â†’ return response.
- `raise HTTPException` for errors, never return error dicts.
- Tests use `httpx.AsyncClient`, not `TestClient`.

## What NOT to Do
- NEVER put business logic in route handlers. Use services.
- NEVER use `db.execute(f"SELECT * WHERE id={id}")`. Use parameterized queries.
- NEVER return 200 for errors. Use proper HTTP status codes.
- NEVER commit `.env` or `alembic.ini` with real credentials.
```

## 5. CLI Tool (Node.js)

```markdown
# my-cli

## Quick Facts
- **Stack**: TypeScript, Node.js 22
- **Build**: `pnpm build` (tsup, single CJS bundle for Node)
- **Test**: `pnpm test` (vitest)
- **Run locally**: `node dist/index.cjs [args]`
- **Link globally**: `pnpm link --global`

## Architecture
- `src/index.ts` â€” Entry point, arg parsing (process.argv or commander)
- `src/commands/` â€” One file per subcommand
- `src/utils/` â€” Shared helpers
- `bin/` â€” Shebang entry (`#!/usr/bin/env node`)

## Conventions
- Exit code 0 for success, 1 for user error, 2 for internal error.
- `process.stderr` for errors and progress. `process.stdout` for output only.
- `--help` and `--version` flags always work.
- Colorized output (chalk/picocolors) with `--no-color` fallback.
- Errors include actionable message: what went wrong + how to fix.

## What NOT to Do
- NEVER use `console.log` for errors. Use `process.stderr.write`.
- NEVER exit without a message on failure.
- NEVER require interactive input without a `--yes` bypass flag.
```

## 6. Monorepo (pnpm workspaces)

```markdown
# my-monorepo

## Quick Facts
- **Structure**: pnpm workspaces
- **Packages**: `@scope/core`, `@scope/web`, `@scope/api`
- **Install**: `pnpm install` (ROOT ONLY, never in packages)
- **Build all**: `pnpm -r build`
- **Test all**: `pnpm -r test`
- **Lint all**: `pnpm -r lint`

## Architecture
- `packages/core/` â€” Shared types, utilities, business logic
- `packages/web/` â€” Next.js frontend
- `packages/api/` â€” Express/Fastify backend
- Root `package.json` â€” workspace config, shared devDependencies

## Conventions
- Shared types ONLY in `@scope/core`. Never duplicate across packages.
- Cross-package imports use the scope: `import { User } from '@scope/core'`
- Each package has its own `tsconfig.json` extending root `tsconfig.base.json`.
- Each package publishes independently with its own version.
- Changes to core require running tests in ALL dependent packages.

## What NOT to Do
- NEVER run `pnpm install` inside a package directory.
- NEVER duplicate types that exist in @scope/core.
- NEVER add devDependencies to individual packages (add to root).
```

## 7. Rust Library

```markdown
# my-crate

## Quick Facts
- **Stack**: Rust (latest stable), no nightly features
- **Build**: `cargo build`
- **Test**: `cargo test`
- **Lint**: `cargo clippy -- -D warnings`
- **Format**: `cargo fmt`
- **Doc**: `cargo doc --open`

## Architecture
- `src/lib.rs` â€” Public API
- `src/` â€” Internal modules
- `tests/` â€” Integration tests
- `examples/` â€” Usage examples (run with `cargo run --example name`)
- `benches/` â€” Benchmarks (run with `cargo bench`)

## Conventions
- No `unwrap()` in library code. Use `?` operator with custom error types.
- Error type in `src/error.rs`. Implement `std::error::Error`.
- All public items have doc comments (`///`).
- Tests inline in each module under `#[cfg(test)] mod tests`.
- Integration tests in `tests/` for public API behavior.
- `#[must_use]` on functions that return values you shouldn't ignore.

## What NOT to Do
- NEVER use `unwrap()` or `expect()` in library code.
- NEVER use `unsafe` without a `// SAFETY:` comment.
- NEVER add dependencies without checking size impact (`cargo bloat`).
```

```markdown
- **Test**: `go test ./...`
- **Lint**: `golangci-lint run`
- **Format**: `gofmt -w .`

- `cmd/server/` â€” Main entry point

- Accept interfaces, return structs.
- Table-driven tests for all functions.
- Error wrapping: `fmt.Errorf("operation failed: %w", err)`

## What NOT to Do
- NEVER ignore errors. Handle or propagate every error.
- NEVER use `panic()` in library code.
- NEVER use `init()` for anything except registering drivers.
```

## 10. Supabase + Next.js

```markdown
# my-app

## Quick Facts
- **Stack**: Next.js 15, Supabase (auth + DB + storage), TypeScript
- **Dev**: `pnpm dev` (frontend) + Supabase local: `supabase start`
- **Build**: `pnpm build`
- **Test**: `pnpm test`
- **DB migrations**: `supabase db push` (local) / `supabase db push --linked` (remote)
- **Generate types**: `supabase gen types typescript --local > src/types/supabase.ts`

## Architecture
- `app/` â€” Next.js App Router
- `supabase/` â€” Migrations, seed data, edge functions
- `lib/supabase/` â€” Client initialization (browser + server)
- `src/types/supabase.ts` â€” Auto-generated DB types

## Conventions
- Server-side Supabase client in Server Components and Server Actions.
- Browser Supabase client in Client Components only.
- ALWAYS regenerate types after schema changes: `supabase gen types typescript`
- Row Level Security (RLS) on EVERY table. No exceptions.
- Edge Functions for server-side logic that needs Supabase access.

## What NOT to Do
- NEVER use the service role key in client-side code.
- NEVER disable RLS, even temporarily.
- NEVER write SQL without a migration file.
- NEVER use browser client in Server Components (use server client).
```
